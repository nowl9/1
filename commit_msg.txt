feat(ledger): run_id/mode stamping + fill-fidelity slippage (paper-trade step 4)

Build step 4 of docs/paper-trading-plan.md (sections 3.4, 6.4).  Makes runs
separable/joinable and adds the fill-fidelity column the analysis needs.

- paper_ledger.py: PaperOrderRecord / PaperFillRecord / PaperSettlementRecord
  / PaperRejectionRecord gain optional run_id + mode (defaulted, so old
  records and direct-construction round-trip at schema_version 1).
  PaperLedger(base_dir, run_id=, mode=) stamps both onto EVERY append via
  model_copy on the frozen record (caller's object untouched).  A bare
  PaperLedger(dir) stamps the field defaults, so existing round-trip
  equality holds.

- main.py: Agent generates a per-run run_id (overridable; goal-2 replay will
  pass a deterministic one) and builds the ledger with run_id + mode (from
  the clock); run_id logged at agent.starting.

- tools/analyze_paper_ledger.py: join now emits a `slippage` column
  (fill_price - limit_price, signed; negative = price improvement; NaN on
  no_fill) alongside the existing realized_pnl / return P&L columns.
  run_id / mode ride through the model_dump into the parquet so runs are
  separable.  slippage added to the empty-frame column set.

Tests:
- test_ledger_run_stamp.py (5): stamping on all four record kinds; stamp is
  authoritative; default ledger round-trips; caller object not mutated;
  schema_version stays 1.
- test_analyze_paper_ledger.py (+6): slippage = fill - limit / zero / NaN;
  slippage + P&L emitted together; run_id/mode survive the join; empty frame
  has slippage.
End-to-end CLI smoke confirmed run_id/mode stamped and slippage in the
parquet.  Full suite 723 passed (712 prior + 11 net new).
